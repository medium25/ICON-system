# Мониторинг результатов

Веб-приложение для отслеживания лидов с тремя уровнями доступа.

## Текущий режим: прототип (без Firebase)

Сейчас все данные (аккаунты, операторы, записи по дням) хранятся прямо в браузере — в `localStorage`. Это значит:

- Всё работает сразу, без регистрации в Firebase — можно открыть `npm run dev` и сразу пользоваться.
- Данные видны только в том браузере, где их ввели — если открыть приложение с другого компьютера или телефона, там будет пусто. Это ограничение прототипа: для реальной многопользовательской работы (операторы заходят каждый со своего устройства) нужна общая база данных.
- Если очистить кэш браузера / localStorage — все данные пропадут.

Когда будете готовы перейти на Firebase (общая база, доступ с любых устройств, реальный хостинг), смотрите раздел [«Переход на Firebase»](#переход-на-firebase) ниже — код для этого уже подготовлен и лежит рядом, нужно просто подставить несколько файлов и настроить проект.

## Роли

- **TOP MANAGER** — полный доступ: главный дашборд, страница «Данные по лидам», страницы всех операторов и тим-лидеров, плюс раздел «Пользователи» для создания новых аккаунтов.
- **TEAM LEADER (тим-лидер)** — видит главный дашборд, вносит данные по лидам (сколько поступило, сколько потрачено) и ведёт свою личную статистику по тем же 4 полям, что и оператор. Аккаунтов этого уровня может быть несколько.
- **CALL-MANAGER (оператор)** — видит и заполняет только свою собственную страницу, ничего больше.

Ролей без пароля больше нет — каждый заходит под своим email/паролем, роль определяется автоматически.

## Запуск прототипа

```bash
npm install
npm run dev
```

Откройте адрес, который выведет Vite. Ни `.env`, ни аккаунт в Firebase не нужны. При первом открытии вместо формы входа появится форма «Создать аккаунт топ-менеджера» — заполните её один раз, дальше все остальные аккаунты создаются уже из-под неё, в разделе «Пользователи».

## Как это устроено

- **`/login`** — вход по email и паролю.
- **Главная (`/`)** — для TOP MANAGER и TEAM LEADER: выбор месяца, плашки LEADS / CPL / SPEND / ACT / VISITED / LOST / NON Q. LEADS, список операторов и тим-лидеров.
- **Данные по лидам (`/#/targetolog`)** — для TOP MANAGER и TEAM LEADER: сколько лидов поступило, сколько денег потрачено за день.
- **Страница оператора/лидера (`/#/operator/<id>`)** — ежедневный ввод: новых лидов взято в работу, некачественные, потерянные, посетили пробное занятие. У CALL-MANAGER это единственная доступная страница (своя); TOP MANAGER и TEAM LEADER могут открыть любую.
- **Пользователи (`/#/admin`)** — только у TOP MANAGER: создание новых аккаунтов (имя, email, пароль, роль) и отключение доступа существующим.

### Формулы на главном дашборде (за выбранный месяц)

- `LEADS` = сумма «лидов поступило» на странице «Данные по лидам»
- `SPEND` = сумма «потрачено»
- `CPL` = `SPEND / LEADS`
- `NON Q. LEADS` / `LOST` / `VISITED` = суммы по всем операторам и тим-лидерам (личная статистика тим-лидера тоже входит в общий подсчёт, так как это тоже обработанные лиды)
- `ACT` (в процессе) = `LEADS − NON Q. LEADS − LOST − VISITED`

Если данных за выбранный месяц нет — плашки покажут 0, это нормально.

# Переход на Firebase

Когда прототип устраивает и нужен доступ с разных устройств (операторы заходят каждый со своего телефона/компьютера), переключаемся на настоящую базу данных. Код для этого уже лежит в проекте — рядом с рабочими файлами есть их Firebase-версии:

| Прототип (сейчас активен) | Firebase-версия (переключиться на неё) |
|---|---|
| `src/lib/firestore.js` | `src/lib/firestore.firebase.js` |
| `src/lib/accounts.js` | `src/lib/accounts.firebase.js` |
| `src/context/AuthContext.jsx` | `src/context/AuthContext.firebase.jsx` |

## Шаг 0. Переключить код на Firebase

```bash
cp src/lib/firestore.firebase.js src/lib/firestore.js
cp src/lib/accounts.firebase.js src/lib/accounts.js
cp src/context/AuthContext.firebase.jsx src/context/AuthContext.jsx
```

## Шаг 1. Создать проект Firebase

1. Зайдите на [console.firebase.google.com](https://console.firebase.google.com) и создайте новый проект (Google Analytics можно не подключать).
2. **Build → Firestore Database** → «Создать базу данных», ближайший регион, режим — **production**.
3. **Build → Authentication** → «Начать» → вкладка **Sign-in method** → включите провайдер **Email/Password**.
4. **Build → Hosting** → «Начать» (деплоить будем через GitHub Actions, вручную ничего загружать не нужно).
5. В настройках проекта (шестерёнка → **Project settings** → **General**) в блоке «Your apps» нажмите «</>» (Web), зарегистрируйте веб-приложение — получите `firebaseConfig` с полями `apiKey`, `authDomain`, `projectId`, `storageBucket`, `messagingSenderId`, `appId`.

## Шаг 2. Локальный запуск

```bash
npm install
cp .env.example .env
# впишите в .env значения из firebaseConfig (шаг 1.5)
npm run dev
```

## Шаг 3. Применить правила безопасности Firestore

**Firestore Database → Rules** → вставить содержимое [`firestore.rules`](firestore.rules) → «Опубликовать».

## Шаг 4. Создать первого TOP MANAGER

Никаких действий в консоли Firebase не требуется. Откройте приложение — если в системе ещё нет ни одного топ-менеджера, вместо формы входа появится форма «Создать аккаунт топ-менеджера» (имя, email, пароль). Заполните её один раз.

Это можно сделать только пока в системе нет ни одного TOP MANAGER — как только первый аккаунт создан, эта форма больше нигде не появляется, и дальше все остальные аккаунты (Call-Manager, Team Leader, дополнительные Top Manager) создаются уже из-под этого аккаунта, в разделе «Пользователи», без обращения к консоли Firebase.

## Шаг 5. Выложить код на GitHub

```bash
git init
git add .
git commit -m "Initial commit"
gh repo create <название-репозитория> --private --source=. --push
```

## Шаг 6. Настроить автоматический деплой (GitHub Actions → Firebase Hosting)

1. В репозитории на GitHub: **Settings → Secrets and variables → Actions → New repository secret**. Добавьте секреты со значениями из `firebaseConfig`:
   - `VITE_FIREBASE_API_KEY`
   - `VITE_FIREBASE_AUTH_DOMAIN`
   - `VITE_FIREBASE_PROJECT_ID`
   - `VITE_FIREBASE_STORAGE_BUCKET`
   - `VITE_FIREBASE_MESSAGING_SENDER_ID`
   - `VITE_FIREBASE_APP_ID`

2. Получите сервисный аккаунт для деплоя: Firebase Console → Project settings → **Service accounts** → «Generate new private key» — скачается JSON. Его **полное содержимое** вставьте как секрет `FIREBASE_SERVICE_ACCOUNT`.

3. Замените `YOUR_FIREBASE_PROJECT_ID` в [`.firebaserc`](.firebaserc) на реальный `projectId` и закоммитьте.

4. `git push` в `main` — сработает [`.github/workflows/deploy.yml`](.github/workflows/deploy.yml) и выложит сайт на Firebase Hosting.

### Про безопасность (после перехода на Firebase)

Правила Firestore (`firestore.rules`) проверяют роль пользователя из документа `users/{uid}`: оператор может читать/писать только свои записи, тим-лидер и топ-менеджер — всё. Создание аккаунтов возможно только из-под TOP MANAGER.

Ограничение клиентского SDK Firebase: «отключить доступ» в разделе «Пользователи» удаляет запись о роли (человек больше не сможет ничего открыть в приложении), но сам логин в Firebase Authentication технически остаётся — полное удаление аккаунта требует Admin SDK/Cloud Functions, что для этого проекта посчитали лишним усложнением. Если понадобится — можно добавить отдельно.

Создание первого TOP MANAGER без консоли Firebase работает через служебный документ `meta/bootstrap`: пока его нет, любой, кто откроет приложение первым, может назначить себя топ-менеджером напрямую с экрана входа. Как только этот документ создан (сразу после первого такого аккаунта), путь самоназначения закрывается навсегда — дальше все роли выдаются только из-под уже существующего TOP MANAGER.

## Как добавлять новых операторов и тим-лидеров

Топ-менеджер заходит в раздел «Пользователи» на главной странице → «Добавить пользователя» → указывает имя, email, пароль и роль. Для Call-Manager и Team Leader автоматически создаётся их личная страница статистики, ссылка на неё появляется на главной.
